# Task Breaker — 태스크 분해 규칙

task-breaker 서브에이전트가 **작업 시작 전 반드시** 읽는 파일입니다.

---

## 1. 원자 태스크 정의

태스크는 다음 조건을 모두 만족해야 합니다:

- **단일 파일 원칙:** 하나의 태스크는 **하나의 파일**만 수정합니다.
  - 예외: 새 파일 생성 + 해당 파일을 등록하는 설정 파일 수정은 묶을 수 있음
- **단일 함수/클래스 원칙:** 한 태스크에서 여러 독립적인 함수를 동시에 구현하면 분리합니다.
- **2개 이상 파일 동시 수정 금지:** 파일 A와 파일 B를 같이 수정해야 한다면 두 개의 태스크로 분리합니다.
- **완료 가능성:** 태스크 하나를 완료하면 테스트 가능한 단위가 되어야 합니다.

---

## 2. plan.json 태스크 필드 규칙

```json
{
  "id": "task_1",
  "title": "태스크 제목 (동사로 시작)",
  "description": "구현 대상과 방법을 구체적으로 서술. 모호한 표현 금지.",
  "target_files": ["수정할 파일 경로"],
  "files_to_create": ["새로 생성할 파일 경로"],
  "dependencies": ["task_0"],
  "parallel_group": 1,
  "priority": "P0|P1|P2",
  "complexity": "low|medium|high",
  "done_criteria": "이 태스크가 완료됐다고 판단하는 구체적 기준"
}
```

### 필드별 규칙

| 필드 | 규칙 |
|------|------|
| `id` | `task_{n}` 형식. n은 1부터 순차 증가. |
| `description` | "~를 구현한다", "~를 수정한다" 형태로 명확하게. "개선" 같은 모호한 표현 금지. |
| `target_files` | 실제 파일 경로. 존재하지 않는 파일이면 `files_to_create`로 이동. |
| `dependencies` | 이 태스크 전에 완료되어야 할 task id 목록. 없으면 `[]`. |
| `parallel_group` | 같은 숫자끼리 동시 실행 가능. 아래 배정 규칙 참고. |
| `done_criteria` | "함수 X가 Y를 반환한다", "테스트 Z가 통과한다" 형태. |

## 3. parallel_group 배정 규칙

```
규칙 1: dependencies가 빈 배열인 태스크는 모두 group=1
규칙 2: group=1의 태스크들이 완료된다고 가정했을 때
         그 결과에만 의존하는 태스크는 group=2
규칙 3: group=2가 완료될 때 의존이 풀리는 태스크는 group=3
         (이하 동일 패턴)
규칙 4: 같은 파일을 수정하는 태스크는 절대 같은 그룹 배정 금지
          → 파일 충돌 방지
```

**예시:**
```
task_1 (group=1): utils.py 생성        ← 의존 없음
task_2 (group=1): models.py 생성       ← 의존 없음
task_3 (group=2): service.py 생성      ← task_1, task_2 의존
task_4 (group=3): api.py 생성          ← task_3 의존
task_5 (group=3): tests/test_api.py 생성 ← task_4 의존
```

## 4. 태스크 수 권고

| 기능 규모 | 권고 태스크 수 | 처리 방법 |
|---------|-------------|---------|
| 소규모 기능 | 3~7개 | 단일 마일스톤 |
| 중규모 기능 | 8~15개 | 마일스톤 2개로 분리 |
| 대규모 기능 | 16개 이상 | 마일스톤 3개 이상으로 묶고 단계별 실행 권고 |

> 한 마일스톤에 7개를 초과하면 인지 부하가 높아집니다. 적극적으로 마일스톤을 나누세요.
